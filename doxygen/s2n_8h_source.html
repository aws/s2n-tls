1052/* Disable TLS intent verification for received certificates.
1053 *
1054 * By default, s2n-tls will verify that received certificates set Key Usage / Extended Key Usage
1055 * fields that are consistent with the current TLS context (e.g. checking that serverAuth is set
1056 * when verifying server certificates as a client, or checking that clientAuth is set when
1057 * verifying client certificates as a server). This verification ensures that received certificates
1058 * are being used for their intended purpose as specified by the issuer.
1059 *
1060 * This verification may be incompatible with some PKIs where intent is improperly specified.
1061 * `s2n_config_disable_x509_intent_verification()` may be called in this case to disable the
1062 * verification. This verification should only be disabled if it is known that all received
1063 * certificates will be issued from a CA that intended for the certificates to be used in the given
1064 * TLS context, despite what's indicated in the Key Usage / Extended Key Usage extensions.
1065 *
1066 * @note If a received certificate doesn't contain a Key Usage / Extended Key Usage extension, it's
1067 * assumed that the issuer permits the certificate to be used for any purpose. s2n-tls will
1068 * only reject a certificate due to invalid intent if the issuer explicitly indicates a
1069 * purpose that is invalid for the TLS context in which it is received.
1070 *
1071 * @param config The associated connection config.
1072 * @returns S2N_SUCCESS on success, S2N_FAILURE on failure.
1073 */
1074S2N_API extern int s2n_config_disable_x509_intent_verification(struct s2n_config *config);
